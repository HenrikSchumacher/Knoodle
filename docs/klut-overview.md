# KLUT — Knot LookUp Table

*First-pass survey, 2026-06-12. Compiled from exploration-agent reads of the
code; items marked ⚠ are second-hand and not yet verified line-by-line by us.*

## Purpose

Identify a knot diagram (≤ 13 crossings after simplification) by exact lookup:
encode the diagram as a **MacLeod code**, pack it into a 128-bit key, and look
it up in a precomputed table mapping keys → knot names like `K[7,5,1,"e/r"]`.

The table was built (Henrik + Jason, May 2026) by:

1. Enumerating candidate diagrams with **plantri** (planar quadrangulation /
   edge-code output → PD codes),
2. Simplifying them with Knoodle (`PlanarDiagramComplex::Simplify` + `Reapr`),
3. Matching simplification endpoints against diagrams generated from
   **KnotInfo exemplar** PD codes, expanded over each knot's symmetry group.

## Key scheme (how a diagram becomes a key)

- `PlanarDiagram::WriteMacLeodCode<UInt8>()` emits one byte per crossing
  (⚠ reported bit layout: upper 6 bits = "leap distance" to the partner
  arc-endpoint, bit 2 = over/under, bit 0 = handedness; verify in
  `src/PlanarDiagram/MacLeodCode.hpp`). Internally a 2n-byte
  "LongMacLeodCode" over all arc endpoints is compressed to n bytes.
- `Klut::Key_T = std::array<UInt64,2>` — 16 bytes, zero-padded, so codes up to
  16 crossings fit; `Klut::max_crossing_count = 13` is the configured limit.
- Packing/unpacking: `src/Klut/Key.hpp` (`MacLeodCodeToKey`,
  `KeyToMacLeodCode`, `MacLeodCodeLength`).
- Hash table: `AssociativeContainer<Key_T, ID_T, Tools::array_hash>` (a Tools
  abstraction; comments show Henrik experimented with `std::unordered_map`,
  `ankerl::unordered_dense`, `boost::unordered_flat_map`).
- **No canonicalization at lookup time.** A raw diagram generally will NOT hit
  the table; the table instead contains *many* MacLeod codes per knot type
  (every code that simplification endpoints actually produced). Coverage
  semantics are an open question — see below.

## Value scheme (what a lookup returns)

- `Klut::FindID(...)` → `pair<Size_T crossing_count, ID_T id>`, where `ID_T =
  UInt16` indexes into the per-crossing-count name table. Sentinels:
  `not_found = 65535`, `error = 65534`, `invalid = 65533` (e.g. multi-component
  input). `FindName(...)` returns the string instead.
- Names look like `K[c, idx, j, "coset"]`:
  - `c` = crossing number, `idx` = index within KnotInfo's `c_idx` naming.
  - `j` — the KnotInfo **alternating** flag (1 = alternating, 0 =
    non-alternating). Stored as integer 0/1 in the raw `K[...]` table name;
    rendered as WL boolean `True`/`False` in `KnotSymbol[...]` form.
    **Correction (2026-06-13):** an earlier note read this as amphichirality —
    that was wrong. `klut_check` showed a knot is identified by `(c,i,j)`
    (`12a_i` and `12n_i` share `i`, differ in `j`), and KnotInfo's column 11 is
    literally "alternating". The Klut.nb `[[All,3]] /. {0->False,1->True}`
    rewrite is just the boolean conversion, regardless of meaning.
  - `"coset"` = which symmetry-group coset of {e, m, r, mr} this entry
    represents (see next section).

## Symmetry handling

From `$PKG/KnoodleLink/KnotInfo.nb`: each KnotInfo knot has a Symmetry Type
that determines which of the 4 variants (e = identity, m = mirror, r =
reverse, mr = both) are distinct knots:

```mathematica
$KnotInfoCosetList = <|
  "reversible"            -> {"e/r", "m/mr"},
  "negative amphicheiral" -> {"e/mr", "m/r"},
  "positive amphicheiral" -> {"e/m", "r/mr"},
  "chiral"                -> {"e", "r", "m", "mr"},
  "fully amphicheiral"    -> {"e/m/r/mr"}
|>
```

So a chiral, non-reversible knot gets 4 table entries; a fully amphichiral
knot (like 4_1) gets 1. Variants are generated by `ChiralityTransform[PD,
mirrorQ, reverseQ]` on the KnotInfo exemplar PD code.

## Generation pipeline (Klutter)

Mathematica orchestration in `$PKG/KnoodleLink/Klutter.nb`; heavy lifting in
C++ `src/Klutter.hpp` via LibraryLink. Per crossing number n:

1. **plantri → PD codes.** A plantri *edge-code* file (header
   `>>edge_code<<`) is imported as a raw string and converted by
   `PlantriCodesToPDCodes` (WL, in Klutter.nb). Diagrams whose traversal has
   more than one component are discarded (`Nothing`) — knots only. The
   resulting code list is randomly permuted so hard cases don't cluster at the
   end of the parallel sieve.
   ⚠ The actual plantri *command line* (which options generated the edge-code
   files) was not found in the notebooks — presumably run by hand. **Open
   question.**
2. **`Klutter[n, threadCount]`** constructed with working/target directories.
3. **`LoadKnotInfo[nameFile, pdCodeFile]`** — loads the exemplar names + PD
   codes (exported from KnotInfo.nb out of `knotinfo_data_complete.tsv`),
   expands symmetry variants, seeds the buckets. Uses Reapr-based random
   embeddings/rotations + simplification to generate many MacLeod codes per
   exemplar (`embedding_trials`, `rotation_trials`; notebook defaults 20/20).
4. **`LoadPlantriPDCodes[codes, embedding_trials, rotation_trials]`** — sieves
   the plantri candidates: simplify each; if the endpoint key is already in
   the table, the candidate's keys join that knot's bucket; equivalences
   discovered along the way merge buckets. Unidentified keys are tracked
   separately (`unidentified_ralt_keys` / `unidentified_nalt_keys` —
   alternating vs non-alternating ⚠).
5. **`ExportKeys[]` / `ExportValues[]`** → `Klut_Keys_NN.bin` /
   `Klut_Values_NN.tsv`. Status checks: `BucketsOkayQ[]`, `SucceededQ[]`,
   `UnidentifiedKeyCount[]`.

## File formats

- `Klut_Values_NN.tsv`: one line per knot-type entry, `name \t key_count`,
  in ID order. Example (`Klut_Values_04.tsv`): `K[4,1,1,"e/m/r/mr"] \t 2`.
- `Klut_Keys_NN.bin`: the keys, concatenated in the same ID order, **n bytes
  per key** (just the MacLeod code, not the padded 16-byte key) — so
  `size(bin) = n × Σ key_counts`. Reader: `src/Klut/Subtable.hpp` (lazy-loads
  each subtable on first use).
